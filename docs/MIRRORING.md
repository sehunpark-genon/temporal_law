# 법제처 사이트 미러링 — 예상 문제점 정리

> "미러링" = law.go.kr 의 페이지(HTML/CSS/JS/이미지)를 통째로 긁어와 우리 쪽에서 그대로 재현·호스팅하는 것.
> 결론부터: **통째 미러링은 비추천.** 아래 제약이 많고, 지금처럼 **텍스트 + 메타데이터를 추출하고 원문은 캐노니컬 URL로 링크백**하는 방식이 훨씬 가볍고 안정적이다.

---

## 1. 렌더링 — "HTML만 긁으면 빈 껍데기"
- 본문·조문·하이퍼링크가 **서버 HTML 에 없음**. 브라우저에서 **JS(EDotXPress)가 그려 넣음**.
- 그냥 `requests` 로 받으면 알맹이가 비어 있어 → **headless 브라우저(Chrome)** 필요 = 느리고 무겁다(법령당 수십 초).
- 본문이 **iframe(lsInfoP.do)** 안에 또 들어있어 통째 캡처가 까다로움.

## 2. 하이퍼링크 — "그대로 가져오면 죽은 링크"
- 링크가 `<a href>` 가 아니라 **`onclick="fncLsLawPop(...)"` 같은 JS 호출**.
- 목적지를 알려면 **링크마다 고유번호(lsJoLnkSeq)로 서버에 다시 질의**해야 함. 미러엔 그 해석 서버가 없으니 **링크가 안 풀림**.
- → 우리는 링크를 **법제처 캐노니컬 URL**(`/법령/{법}/{제N조}`)로 **재작성**해서 해결. 통째 미러였다면 JS·팝업 인프라까지 다 복제해야 동작.

## 3. 정적 리소스 / 경로 / 보안정책
- CSS·JS·폰트·이미지가 `/LSW/`, `/DRF/` 절대경로 → 미러에선 **경로 전부 재작성** 필요.
- **CSP/CORS** 때문에 외부 호스트 리소스가 차단될 수 있음.

## 4. 호출량 / 차단 (rate limit · IP ban)
- 전체 현행법령 **5천+ 건**, 각 본문·조문·링크해석이 **수백 콜** → 전부 미러하면 **요청 폭증**.
- 과호출 시 **속도 제한·IP 차단** 위험. (우리도 호출 0.2s 간격 + 배치로 완화 중)

## 5. 데이터 신선도 (staleness)
- 법은 **계속 개정**됨. 미러는 그 시점 **스냅샷** → 금방 낡음.
- 최신 유지하려면 **변경 감지 + 재크롤** 체계 필요(우리 sync 가 하는 일). 통째 미러는 이 재갱신이 더 무겁다.

## 6. 분량 / 스케일
- 법령 5천+ × (HTML + CSS + JS + 이미지) = **용량 큼**, 빌드·저장·서빙 부담.
- 반면 텍스트+메타만 뽑으면 법령당 수십 KB~MB 수준으로 가볍다.

## 7. 인코딩 / 구조 복잡성
- 한글·특수공백(`\xa0`)·**중첩 리스트(목내용)** 등으로 파싱이 깨지기 쉬움(실제로 버그 한 번 났음).
- 법조문 구조(편·장·절·조·항·호·목, 별표/서식, 부칙, 신구조문, 3단비교)가 복잡 → HTML 미러로는 **의미 구조를 못 살림**. 검색/임베딩엔 **텍스트+메타가 유리**.

## 8. 동적 기능은 복제 불가
- 검색·팝업·연혁·3단비교·위임조문 등은 **서버 로직 의존**. 정적 미러는 이 인터랙션을 못 살림.

## 9. 법적 / 약관 / 저작권
- 법령 콘텐츠 **전체 복제·재배포**는 이용약관·저작권 이슈 소지. **OpenAPI 이용약관 확인 필수.**

## 10. 유지보수 리스크
- 법제처가 사이트/마크업/엔드포인트를 바꾸면 **미러·파서가 깨짐**.
- 특히 우리가 보조로 쓰는 **비공식 내부 엔드포인트(`/LSW/...`)** 는 공지 없이 변경될 수 있음.
