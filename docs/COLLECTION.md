# 법령 본문·하이퍼링크 추출 (수집 코어)

법령(법률) 본문을 가져오면서, 본문 안에 걸려 있는 **모든 하이퍼링크**(다른 법령·시행령·시행규칙·행정규칙·조례·정관 인용/위임)를 추출해 AI팀이 참조할 수 있는 메타데이터(`relations`)로 만든다.

이 문서는 **본문과 하이퍼링크를 어떻게 뽑아 payload 로 만드나**(수집 코어 `collector/`)를 다룬다.
전체 흐름은 **[OVERVIEW.md](OVERVIEW.md)**, 적재 자동화(Temporal/DB)는 **[PIPELINE.md](PIPELINE.md)** 참고.

단발 실행:
```bash
python3 -m collector.core     # 본문 + 하이퍼링크 수집 → output/
python3 -m collector.verify   # "본문의 모든 링크를 다 가져왔나" 검증 리포트
```
> 정관/학칙 추출과 커버리지 검증은 로컬에 설치된 **Chrome** 을 headless 로 사용한다. 없으면 정관만 건너뛰고 나머지는 정상 동작.

---

## 1. 왜 바꿨나 — 기존(3단비교) 방식의 한계

처음엔 본문 JSON + **3단비교 API(`target=thdCmp`)** 로 위임조문을 가져왔다.
3단비교는 이름 그대로 **법률 ↔ 시행령 ↔ 시행규칙** "3단"만 비교하는 기능이라:

- ✅ 시행령(대통령령) · 시행규칙(부령) 위임은 나옴
- ❌ **행정규칙(고시·훈령)**, **조례(자치법규)**, **정관(학칙)** 위임은 안 나옴
- ❌ 다른 법령 **인용**(「형법」 제250조 …)도 안 나옴

즉 본문 화면에 파랗게 걸려 있는 링크의 상당수를 놓쳤다.

## 2. 지금 방식 — 출처별로 가장 적합한 소스 사용

| 링크 종류 | 가져오는 방법 | `resolve_method` |
|---|---|---|
| 시행령 / 시행규칙 위임 | **`lsDelegated` OpenAPI** | `lsDelegated` |
| 행정규칙(고시 등) 위임 | **`lsDelegated` OpenAPI** | `lsDelegated` |
| 자치법규(조례) 위임 | **`lsDelegated` OpenAPI** | `lsDelegated` |
| 정관 / 학칙(공단) 위임 | **Chrome 렌더링 + 팝업 API** | `rendered_dom` |
| 다른 법령 **인용** (「OO법」 제N조) | **본문 텍스트 파싱** | `text_parse` |
| 자기 법령 **내부참조** (제N조) | **본문 텍스트 파싱** | `text_parse` |

### 2-1. 핵심: `lsDelegated` OpenAPI
`http://www.law.go.kr/DRF/lawService.do?target=lsDelegated&MST=...&type=JSON`

3단비교를 대체한다. **한 번의 JSON 호출**로 본문의 모든 위임 링크(시행령·시행규칙·행정규칙·조례)를 조문별로 묶어서 준다. 각 항목에:
- `링크텍스트` — 본문에서 링크가 걸린 단어 (예: `"대통령령"`, `"고시하는"`, `"조례로"`)
- `라인텍스트` — 그 단어가 들어있는 문장
- `조항호목` — 본문 위치 (예: `제2조제10호`)
- 대상 제목 + 대상 일련번호(MST)

HTML 을 긁을 필요가 없어서 깔끔하고 안정적이다.

### 2-2. 정관 / 학칙은 왜 Chrome 인가
정관·학칙(공단 규정)은 `lsDelegated` 에 안 들어온다. 게다가 법제처 본문은 **JS(클라이언트)로 렌더링**돼서 서버 HTML 엔 `<a>` 가 없다. 그래서 Chrome 으로 본문을 렌더링한 뒤, `joDelegatePop(...,'010113',...)` 앵커를 찾아 팝업 API(`conSchlPubRulByLsPop.do`)로 정관 제목을 해석한다.

### 2-3. 인용 / 자기참조는 본문 텍스트 파싱
"다른 법 인용"과 "자기 법 내부참조"는 위임이 아니라서 `lsDelegated` 에 없다. 본문 텍스트에서 직접 찾는다.
- `「형법」 제250조` → 형법 조문 **인용**(`citation`)
- `제11조에 따른 위원회` (「」 없이) → 자기 법령 **내부참조**(`internal_ref`)

#### 법령명 컨텍스트 이어받기
열거식 인용은 법령명을 한 번만 쓴다:
```
「형법」 제2편제24장 살인의 죄 중 제250조, 제252조 및 제254조의 죄
```
→ `「형법」` 한 번 나오면 뒤따르는 `제250조·제252조·제254조` 도 전부 **형법** 조문으로 잡는다.

#### 컨텍스트 만료 / "같은 법"
한 문장에 외부법과 자기 조문이 섞이면 절 경계(`에 따라`, `경우에는` 등)에서 외부법 맥락을 **끊는다**:
```
「아동복지법」 제15조 … 에 따라 … 경우에는 제8조제2항, 제12조 …
   → 제15조 = 아동복지법 인용
   → 제8조·제12조 = 자기 법령(국민기초법) 내부참조
```
단 **`같은 법`** 은 직전 외부법을 그대로 가리키므로 끊지 않는다:
```
「방송법」 제2조제3호에 따른 … 같은 법 제73조제4항 …
   → 제2조, 제73조 모두 방송법 인용
```

## 3. 출력 구조 (`output/`)

| 파일 | 내용 |
|---|---|
| `laws_payload.json` | 전체 법령 통합 배열 |
| `{법령명}_payload.json` | 법령별 단건 |
| `raw/{법령명}_*.json` | 원본 API 응답(검색/본문/위임) |
| `state.json` | 실행 상태 |

각 법령 payload:
```jsonc
{
  "law_name": "장애인복지법",
  "mst": "279697",
  "body": { "format": "text", "content": "…본문…", "article_count": 196 },
  "relations": [ … ],                 // 본문 하이퍼링크들 (출처 조문 오름차순)
  "ordinance_delegations": [ … ],     // 조례는 따로 (아래 5번)
  "relation_stats": {
    "relations_total": 481,
    "by_category": { "법령": 478, "행정규칙": 3 },
    "by_relation_type": { "delegation": 160, "citation": 209, "internal_ref": 112 }
  }
}
```

## 4. `relations` 필드 설명

```jsonc
{
  "relation_type": "delegation",          // 관계 종류: delegation | citation | internal_ref
  "delegation_type": "시행규칙",           // 위임 세부종류 (delegation 일 때만)
  "target_category": "법령",               // 대상 종류: 법령 | 행정규칙 | 자치법규 | 학칙공단 (URL 경로 결정)
  "source_article_no": "제25조의2",        // 이 링크가 있는 '우리 법' 조문 (위치)
  "source_clause": "제25조의2제2항",       // 더 세밀한 위치 (조+항)
  "link_text": "보건복지부령",             // 본문에서 링크가 걸린 실제 문구
  "line_text": "보건복지부령으로 정하는",  // 그 문구가 들어있는 문장(맥락)
  "target_law_name": "장애인복지법 시행규칙", // 연결되는 대상 문서 이름
  "target_article_no": "제2조의4",         // 대상 문서의 조문 (있으면)
  "target_article_title": "인식개선교육의 방법", // 대상 조문 제목
  "target_mst": "282417",                  // 대상 문서 고유번호(MST)
  "target_url": "https://www.law.go.kr/법령/장애인복지법 시행규칙/제2조의4", // 한글 캐노니컬 주소
  "resolve_method": "lsDelegated"          // 알아낸 방법: lsDelegated | text_parse | rendered_dom
}
```

### `relation_type` 3종
| 값 | 뜻 | 예시 |
|---|---|---|
| `delegation` | **위임** — 하위규정에 맡김 | "대통령령으로 정하는" → 시행령 |
| `citation` | **인용** — 다른 법령을 가리킴 | `「형법」 제250조` |
| `internal_ref` | **자기참조** — 같은 법 다른 조문 | "제11조에 따른 위원회" |

### `delegation_type` (위임 세부)
`시행령` · `시행규칙` · `위임행정규칙` · `위임자치법규`(조례) · `위임학칙공단`(정관)

### 빈 칸이 있는 이유
`text_parse`(인용/자기참조)는 본문에서 찾은 거라, 대상 문서를 열어보지 않아 `target_mst`·`target_article_title` 은 비어 있다. `lsDelegated`(위임)는 API가 다 줘서 채워진다.

## 5. 조례(자치법규)는 왜 따로 (`ordinance_delegations`)

본문엔 `조례로` 링크가 **조문당 1개**지만, 클릭하면 그 위임을 이행하는 **전국 지자체 조례 수백 건**이 뜬다. 이걸 수백 개 relation 으로 펼치면 노이즈가 커서, **출처 조문별로 묶고** 개별 조례는 각자의 링크를 가진 리스트로 보존한다.
```jsonc
{
  "source_article_no": "제13조",
  "link_text": "조례로",
  "ordinance_count": 244,
  "ordinances": [
    { "target_law_name": "서울특별시 …복지위원회 조례", "target_url": "https://www.law.go.kr/자치법규/…", "target_mst": "…" },
    …
  ]
}
```
> 코드에 **버전 B(조회용 링크 1개만)** 도 있다(`split_ordinance_lookup`, 주석 처리). 필요하면 전환.

## 6. URL 형식 (한글 그대로)

퍼센트 인코딩(`%EC%9E…`) 없이 사람이 읽을 수 있게 저장한다.

| 종류 | 형식 |
|---|---|
| 법령 | `https://www.law.go.kr/법령/{법령명}/{제N조}` |
| 행정규칙 | `https://www.law.go.kr/행정규칙/{제목}` |
| 자치법규 | `https://www.law.go.kr/자치법규/{조례명}` |
| 학칙공단 | `https://www.law.go.kr/학칙공단/{정관명}` |

> URL 은 **조(條) 단위**까지만 구분한다. 그래서 같은 조문을 여러 호(`제2조제3호`, `제2조제23호`)로 인용해도 같은 조면 **1건으로 합쳐진다**(중복 dedup). 출처 조문·관계종류·대상이 모두 같을 때만 합침.

---

## 7. 정렬
`relations` · `ordinance_delegations` 는 **출처 조문 오름차순**(제1조 → … → 제15조 → 제15조의2 → …)으로 정렬된다.

---

## 8. 알려진 한계
- **인용/자기참조는 텍스트 휴리스틱**이다. `같은 법`·괄호·절 경계 보정을 넣었지만, 한 줄에 외부법 조문과 자기 조문이 특이하게 섞이면 드물게 오귀속될 수 있다. (100% 정확히 하려면 링크마다 법제처 링크해석 API를 호출해야 하는데 링크당 1회라 비용 과다 + 호출 수 제한으로 인해 불가)
- **정관/학칙**은 Chrome 렌더링이 있어야 잡힌다.
- 본문에 링크 문구만 있고 등록된 위임문서가 0건인 **빈 트리거 링크**는 제외한다(누락 아님 — 실제 대상이 없음).
- 조문 네비게이션 중 **항(제N항)** 단위 자기참조는 잡지 않는다(조 단위만).

| 파일 | 역할 |
|---|---|
| `collector/core.py` | 수집 코어 (검색→본문→위임→정관→인용/자기참조→정렬→payload) |

# 부록

## A. 중복 제거(dedup) 기준 — "같은 조에서 같은 대상조"는 1건

dedup 키 = **(관계종류, 대상카테고리, 대상법령, 대상조문, 출처조문)**.
**항(`source_clause`)은 키에 없다.** 즉:

- 같은 출처 조에서 **같은 대상 조**를 여러 항이 인용하면 → **1건** (`source_clause` 엔 첫 번째 항만 남음).
  예) 제25조의3에서 「방송법」 제2조를 제1항(제23호)·제2항(제3호)·제3항(제12호) 세 번 인용 → **방송법 제2조 1건** (source_clause `제25조의3제1항`).
- 같은 출처 조라도 **대상 조가 다르면** → 각각 별도.
  예) 제25조의2 → 시행규칙 제2조의4 / 제2조의5 / 제2조의3 = 3건 유지.

이유: `target_url` 이 **조(條) 단위**라(법령/방송법/제2조), 호만 다른 건 같은 주소 → 합치는 게 맞음.
> 항 단위로 분리하고 싶으면 dedup 키에 `source_clause` 를 넣으면 된다(현재는 조 단위).

## B. `target=thdCmp` (옛 방식, 3단비교) — 예시

`…/DRF/lawService.do?OC=...&target=thdCmp&MST=279697&type=JSON&knd=2` (`knd=2` = 위임조문)

법률 ↔ **시행령** ↔ **시행규칙** 을 나란히 비교한 행 목록을 준다:
```jsonc
{ "LspttnThdCmpLawXService": { "위임조문삼단비교": { "법률조문": [
  {
    "조번호": "0002",
    "조제목": "제2조(장애인의 정의 등)",
    "시행령조문":   { "법령명": "장애인복지법 시행령",   "조번호": "0002",
                     "조제목": "제2조(장애의 종류 및 기준)", "조내용": "①「장애인복지법」… 대통령령으로 정하는 …" },
    "시행규칙조문": { "법령명": "장애인복지법 시행규칙", "조번호": "0002",
                     "조제목": "제2조(장애인의 장애 정도 등)", "조내용": "①「장애인복지법 시행령」… 별표 1 …" }
  }, …  // 행 237개
]}}}
```
**한계**: 구조가 시행령·시행규칙 두 칸뿐 → 행정규칙·조례·정관·인용이 안 나옴. 그래서 `lsDelegated` 로 교체했다.

## C. `target=lsDelegated` (지금 방식) — 예시

`…/DRF/lawService.do?OC=...&target=lsDelegated&MST=276653&type=JSON`

`법령.위임조문정보[]` = 조문별 묶음. 각 조문 = `{ 조정보(어느 조), 위임정보[](그 조가 위임한 것들) }`.
국민기초생활 보장법 제2조 예시:
```jsonc
{
  "조정보": { "조문번호": "2", "조문제목": "정의" },
  "위임정보": [
    { "위임구분": "시행령", "위임법령제목": "국민기초생활 보장법 시행령", "위임법령일련번호": "281585",
```

## D. 정관/학칙 — `joDelegatePop(...,'010113',...)` → `conSchlPubRulByLsPop.do`

정관은 `lsDelegated` 에 없고, 본문이 JS 로 렌더링돼서 서버 HTML 엔 링크가 없다. 그래서 Chrome 으로 본문을 띄운 뒤, 렌더된 DOM 에서 정관 앵커를 찾는다:
```html
<a class="delegatedOrdin_link"
   onclick="joDelegatePop('276653','0015','04','010113','DEG','정관으로 정하는','1121')">정관으로 정하는</a>
```
인자 = `(lsiSeq, joNo, joBrNo, datClsCd, dguBun, lnkText, seq)`.
- `datClsCd='010113'` = **학칙공단(정관)** 코드 (참고: `010102`=행정규칙, `010103`=자치법규)

이 파라미터로 팝업 API 를 호출하면 정관 문서가 나온다:
```
…/LSW/conSchlPubRulByLsPop.do?lsiSeq=276653&joNo=0015&joBrNo=04&datClsCd=010113
   &dguBun=DEG&lnkText=정관으로 정하는&joSeq=1121
→ HTML 응답에서 제목 추출:  "(한국자활복지개발원) 정관"
```
결과 relation:
```jsonc
{ "relation_type": "delegation", "delegation_type": "위임학칙공단", "target_category": "학칙공단",
  "source_article_no": "제15조의4", "link_text": "정관으로 정하는",
  "target_law_name": "(한국자활복지개발원) 정관",
  "target_url": "https://www.law.go.kr/학칙공단/(한국자활복지개발원) 정관",
  "resolve_method": "rendered_dom" }
```
> `conSchlPubRulByLsPop.do` 는 공식 OpenAPI(`/DRF/`)가 아니라 웹사이트 내부(`/LSW/`) 엔드포인트라 OC(인증키) 불필요 · OpenAPI 호출 횟수와 무관.
